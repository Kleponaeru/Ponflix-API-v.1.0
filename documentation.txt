# Anime API Project Context

## Goal

Build a clean, maintainable, production-ready Anime REST API using **Next.js App Router** and **TypeScript**.

The API scrapes data from **Kuramanime** and is intended to be deployed on **Vercel**.

Current source:

* Base URL: `https://v18.kuramanime.ing`

The project emphasizes:

* Clean Architecture
* Single Responsibility Principle
* Reusable parsers
* Strong TypeScript typing
* Minimal duplicated code
* Easy expansion for future endpoints

---

# Tech Stack

* Next.js (App Router)
* TypeScript
* Cheerio (HTML parsing)
* Native fetch API
* Vercel deployment

# Architecture

Every endpoint follows the same flow.

```
API Route
    ↓
Service
    ↓
Kuramanime Client
    ↓
HTTP Request
    ↓
Parser
    ↓
Typed Response
```

Responsibilities:

* Route

  * Read query/path params
  * Validate input
  * Return API response

* Service

  * Fetch HTML from Kuramanime

* Client

  * Generic HTTP GET wrapper
  * Uses BASE_URL

* Parser

  * Convert HTML into typed objects

* Types

  * Shared interfaces only

```

```

Returns:

```
    animeId,
    slug,
    episode,
    path
}
```

Used by every parser instead of regex duplication.

---

## parseNumber()

Location:

```
src/lib/utils/number.ts
```

Purpose:

Converts strings such as:

```

13.690
8.19
```

into numbers:

```
761228
13690
8.19
```

# Coding Standards

Always follow these rules:

* Keep files small and focused.
* One responsibility per file.
* Never duplicate parsing logic.
* Create reusable helper functions when needed.
* Reuse utility functions instead of writing inline regex.
* Keep TypeScript types strict.
* Prefer reusable abstractions over copy-paste.
* Keep API responses consistent using `ok()` and `fail()`.
* Prefer numbers over numeric strings whenever possible.
* Build scalable code suitable for many future endpoints.

---

# Future Refactor
